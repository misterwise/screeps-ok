# screeps-ok

Behavioral conformance test suite for Screeps server implementations. Write a test once, run it against any engine.

**Current status:** 138 tests across movement, combat, actions, structures, room mechanics, and Game API.

| Engine | Passing | Notes |
|--------|---------|-------|
| xxscreeps | 119/138 (86%) | 18 failures are adapter gaps + parity findings |
| vanilla | 133/138 (96%) | 4 failures are adapter timing edge cases |

## Quick Start

```bash
# Clone and install
git clone https://github.com/misterwise/screeps-ok.git
cd screeps-ok
npm install

# Build native addons (skipped during install)
cd node_modules/@screeps/driver && NODE_PATH=../../.. npx webpack && cd ../../..
npx node-gyp rebuild --release -C node_modules/isolated-vm
npx node-gyp rebuild --release -C node_modules/@screeps/driver/native

# Run against xxscreeps
npm test xxscreeps

# Run against vanilla
npm test vanilla
```

## Writing Tests

Tests use [vitest](https://vitest.dev/) with a custom fixture that provides a `shard` object — the adapter instance.

### Minimal test

```typescript
import { describe, test, expect, code } from '../../src/index.js';

describe('creep.move()', () => {
  test('moves one tile in the given direction', async ({ shard }) => {
    // 1. Set up the world
    await shard.createShard({
      players: ['p1'],
      rooms: [{ name: 'W1N1', rcl: 1, owner: 'p1' }],
    });

    // 2. Place objects
    const creepId = await shard.placeCreep('W1N1', {
      pos: [25, 25],
      owner: 'p1',
      body: ['move'],
    });

    // 3. Run player code (returns the last expression)
    const rc = await shard.runPlayer('p1', code`
      Game.getObjectById(${creepId}).move(TOP)
    `);
    expect(rc).toBe(0); // OK

    // 4. Advance the game
    await shard.tick();

    // 5. Inspect results (plain JSON snapshots)
    const creep = await shard.getObject(creepId);
    expect(creep?.kind).toBe('creep');
    if (creep?.kind === 'creep') {
      expect(creep.pos.y).toBe(24);
    }
  });
});
```

### Key concepts

**`createShard`** — sets up the world. Declare players and rooms. Rooms can have terrain, RCL, and an owner.

**`placeCreep` / `placeStructure` / `placeSite` / `placeSource` / `placeMineral`** — add objects to rooms. Returns an ID you can use later.

**`code\`...\``** — tagged template for player code. Safely interpolates IDs and values via `JSON.stringify`. The code runs inside the engine's game runtime with access to `Game`, all Screeps constants (`TOP`, `WORK`, `RESOURCE_ENERGY`, etc.), `RoomPosition`, and `PathFinder`.

**`runPlayer`** — executes the code string in the engine. The last expression is the return value. Returns numbers (action return codes), strings, booleans, objects, or null. Intents generated by the code are processed on the next `tick()`.

**`tick(n?)`** — advances the game by n ticks (default 1). Processes all pending intents.

**`getObject(id)`** — returns a plain JSON snapshot of the object, or null if destroyed. Use `snapshot.kind` to narrow the type (`'creep'`, `'structure'`, `'site'`, `'source'`, `'mineral'`, `'tombstone'`, `'resource'`).

**`findInRoom(room, type)`** — returns all objects of a type in the room. Types: `'creeps'`, `'structures'`, `'constructionSites'`, `'sources'`, `'minerals'`, `'tombstones'`, `'droppedResources'`.

### Tips

- **Return codes**: `0` = OK, `-9` = ERR_NOT_IN_RANGE, `-6` = ERR_NOT_ENOUGH_RESOURCES, `-12` = ERR_NO_BODYPART, `-7` = ERR_INVALID_TARGET. Use the numeric values directly.
- **Store values**: Snapshots omit zero-value store entries. Use `creep.store.energy ?? 0` when checking for empty.
- **Multi-player**: Declare all players in `createShard`. Each `runPlayer` call executes on a single tick. With vanilla, multiple `runPlayer` calls before `tick()` each consume one tick internally.
- **Owned rooms**: Sources in unowned rooms have 1500 capacity (not 3000). Set `rcl` and `owner` on the room to get owned-room behavior.
- **Body parts**: Use lowercase strings: `'work'`, `'carry'`, `'move'`, `'attack'`, `'ranged_attack'`, `'heal'`, `'tough'`, `'claim'`.

## Adapter Selection

The `SCREEPS_OK_ADAPTER` environment variable points to the adapter module. The adapter is loaded once and provides a fresh shard per test via `createShard` + `teardown`.

### xxscreeps adapter
- Uses xxscreeps's `simulate()` API directly (in-process, fast)
- Requires xxscreeps built locally at `../xxscreeps`
- ~0.5s for the full suite

### vanilla adapter
- Wraps `screeps-server-mockup` (spawns engine child processes)
- Requires `npm run build:vanilla` for native addon compilation
- Tests run serially (~6s for the full suite)
- Code is injected via Memory and executed by the bot's main loop

## Known Issues

- **PathFinder CostMatrix routing**: Skipped — needs terrain data in engine's TERRAIN_DATA env key
- **Body part damage order**: Vanilla applies damage to `body[0]` in DB, xxscreeps to `body[last]`. Under investigation — may be a DB storage order difference, not a behavioral difference.
- **Multi-player vanilla**: Second player without an owned room can't execute code reliably
- **Source regeneration**: Engine tick timing for `nextRegenerationTime` needs more investigation
